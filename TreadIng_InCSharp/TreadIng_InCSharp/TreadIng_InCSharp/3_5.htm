کلاس ReaderWriterLock

به طور معمول ، نمونه های یک نوع (شیء) برای عمل خواندن همزمان thread-safe هستند ولی برای update همزمان خیر . این موضوع در مورد منابع ، مانند فایلها نیز صدق می کند . با اینکه یک چاره برای حفاظت از چنین اشیایی از همه نوع دسترسی اعمال قفل بر آنها است ، ولی این کار میتواند به طور نامطلوب مانع از همزمانی مفید شود به خصوص وقتی که تعداد زیادی عملیات read داشته باشیم و تعداد کمی عملیات update . یک نمونه از موردی که چنین اتفاقی در آن می افتد یک برنامه سرویس دهنده اقتصادی است "business application server" که معمولاً داده ها در آن به منظور بازیابی سریع در فیلدهای ایستا (static) کش (cache) میشوند . کلاس ReaderWriterLock برای چنین مواردی و برای ایجاد حداکثر قابلیت قفلگذاری بکار میرود .

 

کلاس ReaderWriterLock دارای متدهای مجزایی برای اخذ قفلهای read و write است (AcquireReaderLock و AcquireWriterLock) . هر دو این متدها یک پارامتر timeout نیاز دارند و در صورتی که timeout تمام شود یک استثناء از نوع ApplicationException ایجاد می کنند ( بر خلاف متدهای مشابه دیگر کلاسهای نخ کشی که در چنین مواردی مقدار false بر میگردانند ) . در صورتیکه یک منبع خیلی مورد درخواست باشد و بر سر آن نزاع باشد ، timeout میتواند رخ دهد .

 

یک قفل از طریق فراخوانی متدهای ReleaseReaderLock و ReleaseWriterLock آزاد میشود . این متدها همچنین قوانین قفلگذاری لانه ای را می شناسند و رعایت میکنند (یعنی میتوانند در قفلگذاری لانه ای بکار روند). یک متد به نام ReleaseLock نیز وجود دارد که تمام قفلها در سطحی از ساختار لانه ای را آزاد میکند . ( سپس میتوان از متد RestoreLock برای قفل کردن مجدد همان سطحی از قفلها که با ReleaseLock آزاد شده بودند استفاده کرد درست مانند رفتار toggling قفل Monitor.Wait  ) .

 

همچنین میتوان قفلگذاری را در ابتدا کار با فراخوانیAcquireReaderLock یک read-lock بوجود آورد و سپس در صورت نیاز با فراخوانی UpgradeToWriterLock آنرا به یک write-lock تبدیل کرد . این متد یک cookie بر میگرداند که در آینده میتواند برای فراخوانی متد DowngradeFromWriterLock از آن استفاده کرد . چنین مکانیزمی به یک شیء reader این امکان را میدهد که به طور موقت دارای write-access شود ضمن اینکه پس از اینکه دوباره به حالت read برگشت مجبور نیست در صف دریافت سرویس قرار گیرد .

 

در مثال زیر چهار ریسمان اجرا شده اند که اولی دائماً آیتمهایی را به یک لیست اضافه میکنند ، دومی دائماً ایتمهایی را از لیست حذف میکند و دوتای دیگر دائماً تعداد عناصر موجود لیست را گزارش میدهند . دوتای اولی قفلهای write-lock اخذ میکنند و دو ریسمان دوم قفلهای read-lock اخذ میکنند . برای هر قفل یک timeout 10 ثانیه ای استفاده میشود (معمولاً اداره کننده های استثناء "exception handlers" را برای بررسی ApplicationException درون کد قرار میدهیم ، لیکن در اینجا بعلت اختصار آنها را حذف کرده ایم  ) .

class Program {

  static ReaderWriterLock rw = new ReaderWriterLock ();

  static List <int> items = new List <int> ();

  static Random rand = new Random ();

  static void Main (string[] args) {

    new Thread (delegate() { while (true) AppendItem(); } ).Start();

    new Thread (delegate() { while (true) RemoveItem(); } ).Start();

    new Thread (delegate() { while (true) WriteTotal(); } ).Start();

    new Thread (delegate() { while (true) WriteTotal(); } ).Start();

  }

  static int GetRandNum (int max) { lock (rand) return rand.Next (max); }

  static void WriteTotal() {

    rw.AcquireReaderLock (10000);

    int tot = 0; foreach (int i in items) tot += i;

    Console.WriteLine (tot);

    rw.ReleaseReaderLock();

  }

  static void AppendItem () {

    rw.AcquireWriterLock (10000);

    items.Add (GetRandNum (1000));

    Thread.SpinWait (400);

    rw.ReleaseWriterLock();

  }

  static void RemoveItem () {

    rw.AcquireWriterLock (10000);

    if (items.Count > 0)

      items.RemoveAt (GetRandNum (items.Count));

    rw.ReleaseWriterLock();

  }

}

از آنجا که افزودن یک ایتم به لیست خیلی سریعتر از حذف ایتم از لیست است در این مثال در متد AppendItem یک SpinWait قرار داده شده است .
