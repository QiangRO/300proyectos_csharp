عواقب متد Thread.Abort Complications with Thread.Abort

تنها راه از سرگیری یک ریسمان انصراف خورده فراخوانی متد ResetAbort نیست . شاید این طور به نظر برسد که با اعمال انصراف ریسمان فوراً پایان می یابد . ولی در واقع اتفاقی که می افتد این است که ریسمان با یک وکیل خوب برای مدتی در حالت مرگ باقی می ماند . در ذیر دلایلی که باعث میشود ریسمان در حالت AbortRequested قرار بگیرد را مشاهده میکنیم :

  • سازنده های ایستای کلاس (static class constructors) هرگز وسط کار انصراف نمیخورند . (زیرا این کار مانند سمی برای ادامه زندگی کلاس در دامنه برنامه خواهد بود )
  • در بلاکهای catch/finally نمیتوان انصراف زد
  • اگر یک ریسمان وقتی که انصراف میخورد در حال اجرای یک کد غیر مدیریت شده باشد اجرایش تا زمانی که به یک کد مدیریت شده برسد ادامه خواهد یافت .

 

آخرین فاکتوری که میتواند بالقوه دردسر ساز باشد این است که .NET Framework خودش در خود اغلب کدهای غیر مدیریت شده (unmanaged) را فراخوانی میکند و گاهاً در آن کدها برای مدت زمان طولانی باقی می ماند . یک نمونه از آن زمانی است که از کلاس شبکه (networking) و یا پایگاه داده (database) استفاده میکنیم . اگر منبع شبکه یا سرویس دهنده بانک اطلاعاتی از کار بیافتند و یا کند جواب دهند ممکن است که اجرا در کد مدیریت نشده برای دقایقی باقی بماند ، بسته به ساختار اجرایی کلاس . در چنین مواردی قطعاً ترجیح نمیدهیم با Join منتظر پایان کار ریسمان شویم ( حداقل متد Join را بدون پارامتر timeout استفاده نمیکنیم ) .

 

اعمال انصراف بر یک کد .NET خالص کمتر دردسر ساز است . به شرطی که بلاکهای try/finally و یا عبارات using با هم همکاری کنند تا تضمین کنند که یک پاکسازی صحیح صورت گرفته و باید یک استثناء ThreadAbortException رخ دهد . هر چند هنوز احتمال خطر وجود دارد . برای مثال کد زیر را در نظر بگیرید :

 

using (StreamWriter w = File.CreateText ("myfile.txt"))

  w.Write ("Abort-Safe?");

 

عبارت using در C# یک اختصار است که میتواند بصورت زیر تشریح شود  :

 

StreamWriter w;

w = File.CreateText ("myfile.txt");

try { w.Write ("Abort-Safe"); }

finally { w.Dispose(); }

 

این امکان وجود دارد که متد Abort پس از ساخته شدن StreamWriter و قبل از شروع بلاک try اعمال شود . در حقیقت با کنکاش در کد IL معادل میتوان مشاهده کرد که حتی امکان دارد Abort پس از ساخته شدن StreamWriter و قبل از انتساب به w رخ دهد :

 

IL_0001: ldstr "myfile.txt"

IL_0006: call class [mscorlib]System.IO.StreamWriter

[mscorlib]System.IO.File::CreateText(string)

IL_000b: stloc.0

.try

{

  ...

 

متد Dispose در بلاک finally نیز به شیوه دیگری یک گذر فرعی بوجود می آورد که حاصل یک دستگیره فایل متروک است که باعث میشود هر تلاشی برای ایجاد فایل myfile.txt تا پایان یافتن دامنه برنامه ناکام بماند .

 

در حقیقت موقعیت این مثال میتواند از این هم بدتر باشد زیرا احتمال زیادی دارد که یک Abort در هنگام ساخته شدن فایل یعنی در زمان کار متد File.CreateText ساخته شود . به این یک کد مبهم گویند که طبق آن ما منبع خود را نخواهیم داشت . خوشبختانه کد .NET هرگز مبهم نخواهد بود . میتوان دوباره سراغ ILDASM برویم و به اسمبلی نگاهی کنیم ، ببینیم که سازنده StreamWriter منطق زیر را دارد :

 

public StreamWriter (string path, bool append, ...)

{

  ...

  ...

  Stream stream1 = StreamWriter.CreateFile (path, append);

  this.Init (stream1, ...);

}

 

در هیچ کجای این سازنده بلاک try/catch وجود ندارد که معنایش این است که اگر Abort در جایی در متد Init رخ دهد stream ساخته شده جدید متروک خواهد شد و هیچ راهی برای بستن دستگیره فایل متناظرش وجود نخواهد داشت .

 

از آنجا که جداسازی (disassembly) هر فراخوانی CLR درخواست شده غیر ممکن است ، این سوال مطرح میشود که چگونه باید متد abort-friendly ساخت . بهترین کلک رشتی که میتوان در این مورد زد این است که اصلاً ریسمان دیگری را Abort نکنیم ، بلکه یک فیلد بولی سفارشی به کلاس ریسمان کارگر اضافه کنیم و با set کردن آن به کارگر اطلاع دهیم که باید Abort کند . ریسمان کارگر به طور دوره ای پرچم را چک می کند و در صورت true بودن به زیبایی خارج میشود . از روی طعنه ، زیباترین خروج برای یک ریسمان کارگر از طریق فراخوانی Abort on itsown thread است هرچند ایجاد یک استثناء بصورت صریح نیز همین کار را خوب انجام میدهد . این کار تضمین میکند که ریسمان پس از گذراندن اجرای بلاکهای catch/finally بدرستی خارج میشود ، تا اندازه ای شبیه فراخوانی Abort از ریسمان دیگری ، با این تفاوت که استثناء فقط در مکانهای مشخص خود رخ میدهد

 

class ProLife {

  public static void Main() {

    RulyWorker w = new RulyWorker();

    Thread t = new Thread (w.Work);

    t.Start();

    Thread.Sleep (500);

    w.Abort();

  }

  public class RulyWorker {

    // The volatile keyword ensures abort is not cached by a thread

    volatile bool abort;

    public void Abort() { abort = true; }

    public void Work() {

      while (true) {

        CheckAbort();

        // Do stuff...

        try { OtherMethod(); }

        finally { /* any required cleanup */ }

      }

    }

    void OtherMethod() {

      // Do stuff...

      CheckAbort();

    }

    void CheckAbort() { if (abort) Thread.CurrentThread.Abort(); }

  }

}

 

 

فراخوانی متد Abort بر خود ریسمان فراخوان شرایطی است که در آن Abort کاملاً امن است . شرایط اینچنینی دیگری نیز زمانی است که مطمئن باشیم ریسمانی را که Abort می کنیم در بخش ویژه ای از کد قرار دارد ، معمولاً بوسیله مکانیزم همزمانی مانند Wait Handle یا Monitor.Wait . سومین مثالی که در آن اعمال Abort کردیم در صورتی ایمن است که متعاقباً پس از اعمال Abort پروسه یا دامنه برنامه ریسمان را متلاشی کنیم (خراب کنیم) .
