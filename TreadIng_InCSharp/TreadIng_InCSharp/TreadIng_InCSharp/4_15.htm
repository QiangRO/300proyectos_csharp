پایان دادن به دامنه های برنامه Ending Application Domains

راه دیگر ساختن یک ریسمان کارگر abort-friendly این است که ریسمانش در دامنه برنامه (application domain) مربوط  به خودش اجرا شود . پس از فراخوانی Abort ، به سادگی دامنه برنامه را متلاشی کنیم . بدین وسیله هر گونه منبعی که بی سرپناه مانده است را آزاد میکنیم .

اگر بخواهیم دقیق شویم ، قدم اول یعنی اعمال Abort بر ریسمان غیر ضروری است زیرا هر زمان که دامنه برنامه متلاشی میشود تمام ریسمانهایی که کدهایی در آن دامنه را اجرا میکنند انصراف میخورند . هر چند زیان اعتماد به این رفتار این است که ریسمان انصراف خورده به موقع خارج نشود ( شاید به دلیل کد درون بلاک finally و یا دلایلی که پیشتر بحث شد ) و لذا دامنه برنامه متلاشی نخواهد شد و در متد فراخوان یک استثناء CannotUnloadAppDomainException رخ خواهد داد .به همین خاطر بهتر است قبل از unload کردن دامنه برنامه ، اول ریسمان کارگر را صریحاً Abort کنیم و سپس متد Join را با یک timeout فراخوانی کرده (تا بر آن کنترل داشته باشیم) .

 

در مثال زیر ، ریسمان کارگر وارد یک حلقه نامحدود میشود ، و یک متد abort-unsafe از File.CreateText میسازد و میبندد . سپس ریسمان اصلی مکرراً ریسمانهای کارگر را به راه میاندازد و Abort میکند ولی معمولاً پس از یکی دو دوره حلقه شکست میخورد زیرا CreateText در داخل بدنه خود Abort شده و پشت یک دستگیره فایل باز رها خواهد شد .

 

using System;

using System.IO;

using System.Threading;

class Program {

  static void Main() {

    while (true) {

      Thread t = new Thread (Work);

      t.Start();

      Thread.Sleep (100);

      t.Abort();

      Console.WriteLine ("Aborted");

    }

  }

  static void Work() {

    while (true)

      using (StreamWriter w = File.CreateText ("myfile.txt")) { }

  }

}

حال برنامه مشابهی را ببینیم که اصلاح شده و ریسمان کارگر در دامنه برنامه خودش اجرا میشود که باعث میشود پس از اینکه ریسمان انصراف خورد unload شود . این برنامه مکرراً و بدون مشکل اجرا میشود زیرا unload کردن دامنه برنامه دستگیره فایل متروک را رها خواهد کرد .

 

class Program {

  static void Main (string [] args) {

    while (true) {

      AppDomain ad = AppDomain.CreateDomain ("worker");

      Thread t = new Thread (delegate() { ad.DoCallBack (Work); });

      t.Start();

      Thread.Sleep (100);

      t.Abort();

      if (!t.Join (2000)) {

        // Thread won't end - here's where we could take further action,

        // if, indeed, there was anything we could do. Fortunately in

        // this case, we can expect the thread *always* to end.

      }

      AppDomain.Unload (ad); // Tear down the polluted domain!

      Console.WriteLine ("Aborted");

    }

  }

  static void Work() {

    while (true)

    using (StreamWriter w = File.CreateText ("myfile.txt")) { }

  }

}

ساخت و تخریب یک دامنه برنامه در دنیای ریسمانها در کلاس یک کار نسبتاً زمانگیر طبقه بندی میشود (چندین میلی ثانیه) لذا بهتر است به جای اینکه درون یک حلقه فراخوانی شود به طور نامنظم انجام گیرد ! همچنین در تفکیک پذیری که در ابتدا توسط دامنه برنامه صورت میگیرد عنصر دیگری معلوم میشود که میتواند مفید یا مضر باشد ، بسته به چیزی که برنامه چند ریسمانی میخواهد بدست آورد . برای مثال در یک زمینه تک تستی ، اجرای ریسمانها بر دامنه های برنامه مجزا میتواند بسیار مفید باشد .
