AutoResetEvent

یک AutoResetEvent بیشتر شبیه یک درب بلیط خوان در ورودی ورزشگاه ( تيرى که چهار بازوى گردنده دارد و هر کس ميخواهد از ان بگذرد کوپن خود را در سوراخ ان انداخته و انرا چرخانده وارد ميشود ، گردان در ) است : وارد کردن یک بلیط اجازه میدهد که دقیقاً فقط یک نفر بگذرد ، واژه "Auto" در نام کلاس اشاره به این مطلب دارد که یک در چرخان باز ، به طور اتوماتیک بسته میشود و یا پس از اینکه به یک نفر اجازه عبور داد "Reset" میشود . یک ریسمان با فراخوانی متد WaitOne (منظور این است که پشت همین در منتظر بمان تا باز شود ) در پشت یک در چرخان منتظر می ماند و یا بلوکه میشود و با فراخوانی متد Set یک بلیط وارد دستگاه میکند . اگر یک تعداد ریسمان متد WaitOne را باهم فراخوانی کنند ، یک صف پشت در چرخان ساخته خواهد شد . یک بلیط میتواند از سوی هر ریسمانی وارد شود – به عبارت دیگر ، هر ریسمانی (ریسمانی که بلوکه نباشد) با دسترسی به یک شیء AutoResetEvent میتواند متد Set آنرا فراخوانی کرده و در نتیجه یک ریسمان بلوکه شده را رها کند .

وقتیکه هیچ ریسمانی در حال انتظار نیست ، اگر متد Set فراخوانی شود ، دستگیره تا زمانیکه ریسمانی متد WaitOne را صدا کند ، باز باقی می ماند . اینگونه رفتار کردن ، کمک می کند تا از نزاع ریسمانها برای سر صف قرار گرفتن جلوگیری شود ، و یک ریسمان با وارد کردن بلیط خود ، متد Set را مکرراً بر یک در چرخان که کسی پشت آن منتظر نیست فراخوانی میکند و بدین صورت اجازه نمیدهد یک جمعیت شلوغ تازه از راه رسیده از میان در عبور کند : فقط تک نفر بعدی مجاز به عبور است و مابقی بلیطها ، باطل "wasted" هستند .

متد WaitOne یک پارامتر timeout اختیاری نیز میپذیرد (اگر از پارامتر استفاده کنیم و انتظار به جای اتمام از طریق دریافت سیگنال ، به علت اتمام زمان tumeout پایان یابد ، متد WaitOne مقدار false برمیگرداند ) . WaitOne همچنین میتواند به منظور خروج از کد همزمانی جاری برای مدت زمان انتظار ، به کار گرفته شود ( در صورتیکه از قفلگذاری اتوماتیک استفاده شود ) تا با این کار از بلوکه شدن مفرط جلوگیری شود .

یک متد به نام Reset نیز تهیه شده است که بدون هیچگونه انتظار یا بلوکه ای در چرخان را میبندد ( البته در باید باز باشد ) .

یک شیء AutoResetEvent میتواند به یکی از دو روش زیر ساخته شود . روش اول این است که از طریق سازنده (constructor) کلاس ساخته شود :

EventWaitHandle wh = new AutoResetEvent (false);

 

اگر پارامتر بولین true باشد ، متد Set دستگیره به طور اتوماتیک بلافاصله پس از ساخت فراخوانی می شود . روش دیگر این است که از طریق کلاس پایه اش نمونه سازی شود ، یعنی کلاس EventWaitHandle :

 

EventWaitHandle wh = new EventWaitHandle (false, EventResetMode.Auto);

 

متد سازنده کلاس EventWaitHandle همچنین اجازه میدهد تا یک ManualResetEvent تولید شود (با تعیین EventResetMode.Manual) .

برای آزاد کردن منابع سیستم عامل وقتی که دیگر مورد نیاز نیستند ، باید متد Close دستگیره انتظار فراخوانی شود . به هر حال ، اگر کسی بخواهد دستگیره انتظار را در کل طول عمر یک برنامه بکار ببرد (همانطور که در اغلب مثالهای این بخش اینگونه است ) ، میتواند از انجام آن صرف نظر کند زیرا این کار به طور اتوماتیک در زمان تخریب دامنه برنامه  (Application Domain) انجام میشود .

در مثال زیر ، یک ریسمان شروع میشود که کارش به سادگی ، انتظار تا زمانی است که سیگنالی از ریسمان دیگری دریافت کند .

class BasicWaitHandle {

  static EventWaitHandle wh = new AutoResetEvent (false);

  static void Main() {

    new Thread (Waiter).Start();

    Thread.Sleep (1000); // Wait for some time...

    wh.Set(); // OK - wake it up

  }

  static void Waiter() {

    Console.WriteLine ("Waiting...");

    wh.WaitOne(); // Wait for notification

    Console.WriteLine ("Notified");

  }

}

 

Waiting... (pause) Notified.

 

 

 

ساختن یک دستگیره انتظار به صورت مستقل از پردازه ( Creating a Cross-Process EventWaitHandle )

 

متد سازنده کلاس EventWaitHandle همچنین این امکان را میدهد که یک EventWaitHandle دارای نام (نامگذاری شده) ساخته شود ، این کار دستگیره را قادر میسازد که در میان چندین پردازه عمل کند . نام ، یک رشته (string) است و میتواند هر مقداری داشته باشد به شرطی که ندانسته با نام دستگیره دیگری یکسان نباشد تا تداخلی پیش نیاید . اگر این نام هم اکنون در کامپیوتر در حال استفاده باشد ، یک ارجاع (نوع reference) به نوع موجود تولید میشود ، در غیر اینصورت ، سیستم عامل یک دستگیره جدید میسازد . مثالی را در اینجا مشاهده میکنیم :

EventWaitHandle wh = new EventWaitHandle (false, EventResetMode.Auto,

"MyCompany.MyApp.SomeName");

 

درصورتیکه دو برنامه هر دو این کد را اجرا کرده باشند ، میتوانند به یکدیگر سیگنال بفرستند : دستگیره انتظار در بین تمام ریسمانهای این دو پردازه کار خواهد کرد .

 

 

تاًیید یا اعلام وصول ( Acknowledgement )

 

فرض کنید مایلیم کارها را در پس زمینه انجام دهیم ، بدون اینکه با دریافت هر کار جدید سرباری را به خاطر ایجاد یک ریسمان جدید بوجود آوریم . ما میتوانیم این کار را با یک ریسمان کارگر که دائماً در یک حلقه میچرخد انجام دهیم (انتظار برای یک کار ، انجام آن کار ، و سپس انتظار برای کار بعدی ) . این یک سناریوی چند ریسمانی مرسوم است . علاوه بر کاهش سربار در ساختن ریسمان ، اجرای وظایف و کارها نیز پشت سر هم صورت میگیرد و در نتیجه پتانسیل فعل و انفعال ناخواسته بین چند ریسمان کارگر و مصرف افراطی منبع را از بین می برد .

ما باید تصمیم بگیریم که چه کاری میخواهیم انجام دهیم ، به هر حال ممکن است با رسیدن یک کار جدید ، ریسمان کارگر مشغول انجام کار قبلی باشد . فرض کنید در این حالت تصمیم میگیریم تا ریسمان فراخوان را تا تکمیل کار قبلی بلوکه کنیم . چنین سیستمی میتواند با دو شیء AutoResetEvent پیاده سازی شود : یک شیء AutoResetEvent به نام "ready" که توسط ریسمان کارگر ، زمانیکه در حالت آماده (آماده انجام کار - بیکار ) است Set میشود و یک شیء AutoResetEvent دیگر به نام "go" که توسط ریسمان فراخوان ، زمانیکه کار جدیدی وجود دارد ، Set میشود . در مثال زیر ، یک فیلد رشته ای ساده برای توصیف کار استفاده شده است (این فیلد با کلمه کلیدی volatile تعریف شده است تا تضمین کند که هر دو ریسمان همیشه نسخۀ یکسانی را می بینند ) :

 

class AcknowledgedWaitHandle {

  static EventWaitHandle ready = new AutoResetEvent (false);

  static EventWaitHandle go = new AutoResetEvent (false);

  static volatile string task;

  static void Main() {

    new Thread (Work).Start();

    // Signal the worker 5 times

    for (int i = 1; i <= 5; i++) {

      ready.WaitOne(); // First wait until worker is ready

      task = "a".PadRight (i, 'h'); // Assign a task

      go.Set(); // Tell worker to go!

    }

    // Tell the worker to end using a null-task

    ready.WaitOne(); task = null; go.Set();

  }

  static void Work() {

    while (true) {

      ready.Set(); // Indicate that we're ready

      go.WaitOne(); // Wait to be kicked off...

      if (task == null) return; // Gracefully exit

      Console.WriteLine (task);

    }

  }

}

برای اینکه به ریسمان کارگر بگوییم از حلقه خارج شود و در واقع کارش تمام شود یک کار null به عنوان سیگنال به او میدهیم . فراخوانی Interrupt یا Abort بر ریسمان کارگر در این مورد به طور مشابه خوب کار خواهد کرد ( از آنجا که ابتدا ready.WaitOne را صدا کرده ایم ) . چرا که پس از فراخوانی ready.WaitOne میتوانیم از موقعیت ریسمان کارگر مطمئن باشیم (چه این کار را – فراخوانی ready.WaitOne – را در عبارت go.WaitOne انجام دهیم و چه درست قبل از آن ) و لذا با این کار از پیچیدگی حاصل از اعمال وقفه بر کد جلوگیری میکنیم . برای فراخوانی Interrupt یا Abort ، همچنین باید استثناء که بسیار مهم است را نیز در ریسمان کارگر در نظر بگیریم و مدیریت کنیم .

 

 

صف تولید کننده / مصرف کننده ( Producer/Consumer Queue )

 

یک سناریوی ریسمانی مرسوم دیگر نیز استفاده از یک پردازه کارگر پس زمینه است که وظایف خود را از یک صف میگیرد . نام این صف ، صف تولیدکننده / مصرف کننده است : تولیدکننده کارها را به صف وارد میکند و مصرف کننده کارها را از صف خارج میکند . این کار تا اندازه ای شبیه مثال قبل است ، به جز اینکه در اینجا اگر ریسمان کارگر مشغول انجام کاری باشد ،  فراخواننده (caller) بلوکه نمیشود .

صف تولید کننده / مصرف کننده قابلیت گسترش و بزرگ شدن را دارد به طوری که در آن چندین مصرف کننده میتواند ساخته شود که هر کدام از آنها صف یکسانی را سرویس میدهد ولی در قالب چند ریسمان مجزا . این کار روش مناسبی برای بهره کشی (نوعی سوء استفاده ، استفاده به صورتی که سودمندی آن با زیانی برای کسی همراه است) از سیستمهای چند پردازنده است بدین صورت که تعداد ریسمانهای کارگر را محدود کرده ایم تا دچار مشکلات ناشی از فراوانی ریسمانها نشویم (یعنی تعویض متن یا context switching بیش از حد و نزاع بر سر منبع ) .

در مثال زیر از یک شیء AutoResetEvent برای اعمال سیگنال (آگاه کردن) به ریسمان کارگر استفاده شده است که فقط زمانی که بیکار میشود در حالت انتظار است (یعنی زمانی که صف خالی باشد ) . برای پیاده سازی صف از یک مجموعه عمومی (public ) استفاده شده است ، که دسترسی به آن برای تضمین thread-safety باید با یک Lock محافظت شود. برای پایان یافتن کار ریسمان کارگر یک کار null وارد صف می کنیم :

 

using System;

using System.Threading;

using System.Collections.Generic;

class ProducerConsumerQueue : IDisposable {

  EventWaitHandle wh = new AutoResetEvent (false);

  Thread worker;

  object locker = new object();

  Queue<string> tasks = new Queue<string>();

  public ProducerConsumerQueue() {

    worker = new Thread (Work);

    worker.Start();

  }

  public void EnqueueTask (string task) {

    lock (locker) tasks.Enqueue (task);

    wh.Set();

  }

  public void Dispose() {

    EnqueueTask (null); // Signal the consumer to exit.

    worker.Join(); // Wait for the consumer's thread to finish.

    wh.Close(); // Release any OS resources.

  }

  void Work() {

    while (true) {

      string task = null;

      lock (locker)

      if (tasks.Count > 0) {

        task = tasks.Dequeue();

        if (task == null) return;

      }

      if (task != null) {

        Console.WriteLine ("Performing task: " + task);

        Thread.Sleep (1000); // simulate work...

      }

      else

        wh.WaitOne(); // No more tasks - wait for a signal

    }

  }

}

 

Here's a main method to test the queue:

class Test {

  static void Main() {

    using (ProducerConsumerQueue q = new ProducerConsumerQueue()) {

      q.EnqueueTask ("Hello");

      for (int i = 0; i < 10; i++) q.EnqueueTask ("Say " + i);

      q.EnqueueTask ("Goodbye!");

    }

    // Exiting the using statement calls q's Dispose method, which

    // enqueues a null task and waits until the consumer finishes.

  }

}
