قفل گذاری و امنیت نخ ( Locking and Thread Safety )

مکانیزم قفل گذاری ، یک دسترسی انحصاری را بوجود می آورد ، و برای این بکار میرود تا تضمین کند فقط یک ریسمان در یک زمان میتواند وارد بخشهای ویژه ای از کد اجرایی شود. برای مثال کلاس زیر را در نظر بگیرید :

class ThreadUnsafe {

  static int val1, val2;

  static void Go() {

    if (val2 != 0) Console.WriteLine (val1 / val2);

    val2 = 0;

  }

}

این کد یک کد thread-safe نیست : اگر متد Go به طور همزمان توسط دو ریسمان فراخوانی شود ، احتمال بوجود آمدن خطای تقسیم بر صفر وجود دارد – از آنجا که مقدار متغیر val2 میتواند در یک ریسمان صفر شود در حالیکه درست در همین لحظه ریسمان دیگر مشغول اجرای کد بین عبارت شرطی if و console.writeline است.

حال ببینیم Lock ( قفلگذاری ) چگونه این مشکل را حل می کند :

class ThreadSafe {

  static object locker = new object();

  static int val1, val2;

  static void Go() {

    lock (locker) {

      if (val2 != 0) Console.WriteLine (val1 / val2);

      val2 = 0;

    }

  }

}

درهر زمان فقط یک ریسمان میتواند شیء همزمانی ( در اینجا Locker ) را قفل کند ، و هر ریسمان مدعی و رقیب تا زمانی که قفل باز شود ، بلوکه میشود . اگر بیش از یک ریسمان مدعی یک قفل باشند ، در یک صف FIFO ( صف آماده ) صف بندی می شوند و گارانتی میشود که  به محض موجود بودن قفل آن را در اختیار عنصر سر صف قرار دهد . قفلهای انحصاری گاهاً بعنوان عامل ایجاد دسترسی نوبتی به آنچه قفل از آن محافظت میکند شناخته میشوند. زیرا دسترسی یک ریسمان نمی تواند با ریسمان دیگری همپوشانی داشته باشد ( همزمان باشد ) . در مورد این مثال ، ما منطق درون متد Go و همچنین متغیرهای val1 و val2 را محافظت کرده ایم.

یک ریسمان بلوکه شده در حالیکه مدعی و در انتظار یک قفل است دارای مقدار WaitSleepJoin برای ویژگی ThreadState خود است. بعداً در مورد اینکه چگونه یک ریسمان بلوکه شده در این حالت میتواند توسط ریسمانهای دیگر که متد Intrrupt یا Abort آن را فراخوانی میکنند ، از حالت بلوکه خارج شود ، صحبت خواهیم کرد. این کار انصافاً یک تکنیک سخت و سنگین ( همان خفن ) است که ممکن است معمولاً برای پایان دادن به یک ریسمان کارگر بکار رود .

 

عبارت Lock در C# در واقع یک واژۀ میانبر ( جایگزین ) برای فراخوانی متدهای Monitor.Enter و Monitor.Leave است که در یک بلاک try-finally فراخوانی شوند. در اینجا رویداد واقعی که در متد Go در مثال قبل اتفاق می افتد را میبینیم :

 

try {

  Monitor.Enter (locker);

  if (val2 != 0) Console.WriteLine (val1 / val2);

  val2 = 0;

}

finally { Monitor.Exit (locker); }

 

فراخوانی متد Monitor.Exit بدون اینکه قبلاً Monitor.Enter را برای شیء مورد نظر فراخوانده باشیم ، باعث رخ دادن یک استثناء میشود.

شیء Monitor همچنین یک متد به نام TryEnter دارد که اجازه میدهد یک timeout تعریف کنیم ( بصورت میلی ثانیه یا TimeSpan ) . این متد ( TryEnter ) در صورتی که یک قفل بدست آید ( قفلی که قبلاً توسط ریسمان دیگری قفل شده است و حالا مورد درخواست است ) مقدرا true و در صورتی که timeout تمام شود و هنوز قفل بدست نیاید ( آزاد نشود ) مقدار false برمیگرداند.متد TryEnter همچنین میتواند به منظور قابل دسترس بودن یک قفل بصورت بدون پارامتر فراخوانی شود ، در اینصورت به محض فراخوانی timrout تمام میشود ، اگر قفل باز باشد معلوم میشود که قابل دسترس است و اگرنه که قفل است و بدین صورت میتوان قفلی را تست کنیم.
