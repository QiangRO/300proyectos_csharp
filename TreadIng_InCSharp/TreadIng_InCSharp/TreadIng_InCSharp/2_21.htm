متون همزمانی ( Synchronization Contexts )

علاوه بر قفلگذاری دستی (manual) می توان به شکل اعلانی نیز قفلگذاری کرد . می توان با اشتقاق از کلاس ContextBoundObject و بکار بردن خاصیت Synchronization ، با CLR قفلگذاری اتوماتیک انجام داد . در اینجا مثالی را مشاهده می کنیم :

using System;

using System.Threading;

using System.Runtime.Remoting.Contexts;

[Synchronization]

public class AutoLock : ContextBoundObject {

  public void Demo() {

    Console.Write ("Start...");

    Thread.Sleep (1000); // We can't be preempted here

    Console.WriteLine ("end"); // thanks to automatic locking!

  }

}

public class Test {

  public static void Main() {

    AutoLock safeInstance = new AutoLock();

    new Thread (safeInstance.Demo).Start(); // Call the Demo

    new Thread (safeInstance.Demo).Start(); // method 3 times

    safeInstance.Demo(); // concurrently.

  }

}

CLR تضمین می کند که در یک زمان فقط  یک ریسمان میتواند کد درون  safeInstance را اجرا کند . CLR این کار با با ساختن یک شیء همزمانی انجام می دهد ( که آنرا با فراخوانی هر یک از متدها یا خصوصیات شیء safeInstance قفل می کند ) . دامنه یا قلمرو یا ناحیه ای که قفل میشود ( در اینجا شیء safeInstance ) یک متن همزمانی یا  "Synchronization context"نامیده میشود .

حال این مکانیزم چگونه عمل می کند ؟ یک سرنخ یا راهنما در فضای نام (namespace) ویژگی Synchronization است : یعنی System.Runtime.Remoting.Context . یک ContextBoundObject میتواند بعنوان شیء "remote" شناخته شود که معنی آن ، این است که تمام فراخوانیهای متدها کنترل میشوند (استراق سمع میشوند) . برای انجام این استراق سمع ، وقتی که یک شیء AutoSafe را نمونه سازی می کنیم ، در واقع CLR یک نماینده یا وکیل برمیگرداند (یک شیء با ویژگیها و متدهای یک شیء AutoSafe که بعنوان یک واسطه عمل می کند ) . از طریق همین واسطه است که قفلگذاری اتوماتیک انجام می شود . رویهم رفته ، استراق سمع در تقریباً یک میکروثانیه به هر فراخوانی متدی افزوده میشود .

 

همزمانی اتوماتیک نمیتواند برای حفاظت از عناصر ایستای کلاس (static type members) بکار رود .نه کلاسها و نه مشتقات کلاس ContextBoundObject (مانند فرم ویندوزی) .

 

قفلگذاری به شیوه مشابهی بصورت داخلی انجام میشود . شاید توقع داشته باشید مثال زیر نیز نتیجۀ مشابهی مانند قبلی داشته باشد :

[Synchronization]

public class AutoLock : ContextBoundObject {

  public void Demo() {

    Console.Write ("Start...");

    Thread.Sleep (1000);

    Console.WriteLine ("end");

  }

  public void Test() {

    new Thread (Demo).Start();

    new Thread (Demo).Start();

    new Thread (Demo).Start();

    Console.ReadLine();

  }

  public static void Main() {

    new AutoLock().Test();

  }

}

( توجه داشته باشید که در اینجا از Console.WriteLine ) استفاده کرده ایم . از آنجا که با یک شیء از نوع این کلاس فقط یک ریسمان در هر زمان میتواند کد را اجرا کند ، هر سه ریسمان جدید در متد Demo بلوکه میشوند تا زمانیکه متد Test پایان یابد (که پایان آن نیز مستلزم این است که ReadLine کامل شود ) . لذا نتیجه این برنامه مانند نتیجه مثال قبل خواهد شد ولی پس از فشردن کلید Enter . چنین thread-safety تقریباً آنقدر بزرگ است که از هرگونه چند ریسمانی مفید در کلاس جلوگیری میکند .

بنابراین ، ما مشکل را حل نکرده ایم : برای مثال اگر AutoLock یک کلاس مجموعه ای (collection class) بود ، هنوز برای اینکه کنترل اجرای آن از طریق کلاس دیگری را در دست بگیریم احتیاج داشتیم تا یک قفل بر روی عبارتی مانند زیر بگذاریم:

if (safeInstance.Count > 0) safeInstance.RemoveAt (0);

 

مگر اینکه کلاس این کد ، خود یک ContextBoundObject همزمان شده (synchronized) باشد !

 

متن همزمانی میتواند در حد قلمرو یک شیء تنها باشد . به طور پیش فرض ، اگر یک شیء همزمان شده در داخل کد شیء دیگری (کلاس دیگری) نمونه سازی شود (instantiate) ، هر دو شیء یک متن همزمانی یکسانی را به اشتراک میگذارند (به عبارت دیگر ، یک قفل بزرگ خواهیم داشت) . چنین رفتاری میتواند با تعیین مقدار یک پرچم Integer در متد سازنده ویژگی همزمانی (SynchronizationAttribute) تغییر کند . این مقدار میتواند یکی از ثابتهای زیر باشد که در کلاس SynchronizationAttribute تعریف شده اند :

 

Constant

Meaning

NOT_SUPPORTED

Equivalent to not using the Synchronized attribute

SUPPORTED

Joins the existing synchronization context if instantiated from another

synchronized object, otherwise remains unsynchronized

REQUIRED (default)

Joins the existing synchronization context if instantiated from another

synchronized object, otherwise creates a new context

REQUIRES_NEW

Always creates a new synchronization context

 

بنابراین اگر شیء ای از کلاس SynchronizedA شیء دیگری از کلاس SynchronizedB را نمونه سازی کند و کلاس SynchronizedB بصورت زیر تعریف شده باشد ، دارای متون همزمانی مجزایی خواهند بود :

 

[Synchronization (SynchronizationAttribute.REQUIRES_NEW)]

public class SynchronizedB : ContextBoundObject { ...

 

هر چه دامنه متن همزمانی بزرگتر باشد ، مدیریت و اداره آن ساده تر است ولی در این صورت شانس همزمانی مفید کمتر میشود . در طرف دیگر این قضیه (یعنی تغییر اندازه دامنه متن همزمانی) ، با مجزا کردن متون همزمانی ، زمینه را برای بوجود آمدن بن بست (deadlock) مستعد کرده ایم . مثالی را در اینجا می بینیم :

 

[Synchronization]

public class Deadlock : ContextBoundObject {

  public DeadLock Other;

  public void Demo() { Thread.Sleep (1000); Other.Hello(); }

  void Hello() { Console.WriteLine ("hello"); }

}

public class Test {

  static void Main() {

    Deadlock dead1 = new Deadlock();

    Deadlock dead2 = new Deadlock();

    dead1.Other = dead2;

    dead2.Other = dead1;

    new Thread (dead1.Demo).Start();

    dead2.Demo();

  }

}

از آنجا که هر نمونه ای از کلاس Deadlock در کلاس Test ساخته می شود (که Test یک کلاس همزمان نشده است) ، هر نمونه ، متن همزمانی مربوط به خود و بنابراین قفل مربوط به خود را خواهد داشت . وقتی دو شیء یکدیگر را فراخوانی می کنند ، زمان زیادی طول نمی کشد که بن بست رخ میدهد (به طور دقیقتر یک ثانیه) . این مشکل زمانی جدی تر یا موزیانه تر میشود که کلاسهای Deadlock و Test توسط تیمهای برنامه نویسی متفاوتی نوشته شوند . نمیتوان از تیم طراحی کلاس Test توقع داشت که کلاس را طوری طراحی کنند که حتی از تجاوزات و تخلفات خود نیز آگاه باشد (منظور این است که فاقد هر گونه اشتباهی باشد) .
