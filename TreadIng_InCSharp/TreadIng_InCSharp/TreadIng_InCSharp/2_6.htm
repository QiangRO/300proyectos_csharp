انتخاب شیء همزمانی ( Choosing the Synchronization Object )

هر شیء ای که برای همه ریسمانهای شریک ( ریسمانهایی که ناحیه ای از کد بین آنها مشترک است ) قابل رویت باشد میتواند بعنوان یک شیء همزمانی به کار رود. ولی باید تابع یک قانون سخت باشد : باید یک نوع reference باشد . همچنین شدیداً توصیه میشود که شیء همزمان کننده یک scope خصوصی ( private ) برای کلاس داشته باشد ( یعنی یک فیلد نمونه private باشد ) تا از هرگونه فعل و انفعال احتمالی غیر عمد از سوی یک شیء یکسان جلوگیری شود ( زیرا آن شیء نمونه نیز چنین قفلگذاری را برای خود انجام میدهد ) . با تبعیت از این قوانین ، شیء همزمانی میتواند بعنوان شیءای که حفاظت میکند دو برابر شود مانند فیلد list در زیر :

class ThreadSafe() {

  List <string> list = new List <string>();

  void Test() {

    lock (list) {

      list.Add ("Item 1");

      ...

به طور معمول یک فیلد اختصاصی برای همزمانی به کار میرود ( مانند Locker در مثال قبل ) ، زیرا امکان کنترل دقیق بر scope قفل را فراهم می کند . استفاده از خود شیء یا نوع ، بعنوان شیء همزمانی : یعنی

lock (this) { ... }

or :

lock (typeof (Widget)) { ... } // For protecting access to statics

 

دلسرد کننده ( نامناسب ) است ، زیرا این کار بالقوه یک scope عمومی ( public ) به شیء همزمانی میدهد.

 

عمل قفلگذاری خودش ذاتاً به هیچ وجه دسترسی به یک شیء همزمانی را محدود نمی کند. به بیان دیگر ، x.ToString() به خاطر اینکه ریسمان دیگری Lock(x) را فراخوانی کرده بلوکه نمیشود – برای اینکه بلوکه کردن به طور کامل اتفاق بیافتد ، لازم است هر دو ریسمان Lock(x) را متناوباً فراخوانی کنند.
