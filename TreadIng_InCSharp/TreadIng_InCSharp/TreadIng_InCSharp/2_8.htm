چه موقع قفل گذاری کنیم ( When to Lock )

بعنوان یک قانون پایه ای ، هر فیلد قابل دسترس برای چندین ریسمان باید در قالب یک قفل خوانده ( read ) و نوشته ( write ) شود . حتی در ساده ترین مورد – مثلاً یک عمل اختصاص مقدار به یک فیلد تنها – باید همزمانی را در نظر گرفت. در کلاس زیر نه متد Increment و نه متد Assign هیچ کدام Thread-safe نیستند.

class ThreadUnsafe {

  static int x;

  static void Increment() { x++; }

  static void Assign() { x = 123; }

}

در زیر ورژنهای thread-safe متدهای Increment و Assign را می بینیم :

class ThreadUnsafe {

  static object locker = new object();

  static int x;

  static void Increment() { lock (locker) x++; }

  static void Assign() { lock (locker) x = 123; }

}

به عنوان یک راه چاره جایگزین برای قفلگذاری در این مورد ساده ، میتوان از ساختار همزمانی غیر – بلوکه (non-blocking ) استفاده کرد. که در بخش 4 توضیح داده خواهد شد .

 

 

قفل گذاری و اتمی عمل کردن ( Locking and Atomicity )

 

اگر گروهی از متغیرها همیشه در یک قفل یکسان نوشته و خوانده شوند میتوان گفت این متغیرها بصورت اتمی خوانده و نوشته می شوند. فرض کنید فیلدهای x و y ، فقط در قفلی که بر شیء locker گذاشته شده خوانده و اختصاص دهی ( نوشته ) میشوند :

lock (locker) { if (x != 0) y /= x; }

 

گفته میشود که x و y بصورت اتومیک دسترسی میشوند زیرا این تکه از کد برنامه نمیتواند تقسیم شود ( بشکند یا مشترک شود ) یا با عملکرد ریسمان دیگری به انحصار درآید ، لذا مقادیر x و y را نخواهد توانست تغییر دهد و لذا خروجی این تکه از کد هرگز اشتباه نخواهد بود. شما هرگز خطای تقسیم بر صفر را نخواهید داشت ، زیرا x و y همیشه در یک قفل انحصاری یکسان دسترسی میشوند.
