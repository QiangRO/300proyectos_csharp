فرستادن داده به تابع ThreadStart ( Passing Data to ThreadStart )

میخواهیم در مثال فوق خروجی ریسمانها را از یکدیگر تفکیک کنیم ، مثلاً یکی hello را با حروف بزرگ بنویسد . این کار را می توانیم با فرستادن یک flag به متد Go انجام دهیم : لذا دیگر نمیتوانیم از نماینده ThreadStart استفاده کنیم زیرا آرگومان ورودی نمیپذیرد. خوشبختانه .NET framework یک نماینده دیگر به نام ParameterizedThreadStart برای اینکار دارد که یک شی بعنوان آرگومان ورودی میپذیرد.

 

public delegate void ParameterizedThreadStart (object obj);

 

لذا مثال قبل این بار اینگونه خواهد شد :

class ThreadTest {

  static void Main() {

    Thread t = new Thread (Go);

    t.Start (true); // == Go (true)

    Go (false);

  }

  static void Go (object upperCase) {

    bool upper = (bool) upperCase;

    Console.WriteLine (upper ? "HELLO!" : "hello!");

  }

 

hello!

HELLO!

 

 در این مثال کامپایلر به طور اتوماتیک ParameterizedThreadStart را تشخیص میدهد زیرا متد Go یک شیء برای پارامتر ورودی دارد. ما همچنین میتوانیم کد را بصورت زیر بنویسیم :

 

Thread t = new Thread (new ParameterizedThreadStart (Go));

t.Start (true);

 

یک جنبه از استفاده ParameterizedThreadStart این است که ما باید آرگومان شیء را قبل از استفاده به نوع مناسبش (در اینجا bool ) تفییر دهیم. همچنین فقط یک الگوی تک آرگومانی از این نماینده موجود است.

یک راه دیگر ، اگر نخواهیم از ParameterizedThreadStart استفاده کنیم ، این است که از یک متد گمنام به شکل زیر استفاده کنیم :

static void Main() {

  Thread t = new Thread (delegate() { WriteText ("Hello"); });

  t.Start();

}

static void WriteText (string text) { Console.WriteLine (text); }

 

فایده این کار این است که متد هدف (در اینجا WriteText) میتواند هر تعداد آرگومان بپذیرد ، و همچنین تعویض و تصحیح نوع لازم نیست. ولی به هر حال هر کس میتواند متغیرهای خارجی متدهای گمنام را تغییر دهد ، همانگونه که در مثال زیر مشاهده میکنید :

static void Main() {

  string text = "Before";

  Thread t = new Thread (delegate() { WriteText (text); });

  text = "After";

  t.Start();

}

static void WriteText (string text) { Console.WriteLine (text); }

 

After

 

در استفاده از متدهای گمنام یک نکته قابل توجه این است که اگر پس از اینکه کار ریسمان آغاز شد یعنی متد start آن فراخوانی شد ، متغیرهای بیرونی (بیرونی از نگاه ریسمان ، مانند text در مثال فوق) را دستکاری کنیم این احتمال عجیب و غریب و مضحک را بوجود آورده ایم که با ریسمان ، فعل و انفعال ناخواسته انجام دهیم. فعل و انفعال مطلوب با ریسمان ( که معمولاً با استفاده از آرگومانها و فیلدهای ورودی متدهای ریسمانها – مثل text در مثال فوق - انجام میشود ) به طور کلی بیش از حد کافی استفاده میشوند. زمانی که یکبار اجرای ریسمان آغاز میشود ، متغیرهای بیرونی در صورتی که read-only باشند بهتر عمل میکنند. مگر اینکه کسی مایل باشد یک مکانیزم قفل گذاری دقیق بر هر دو سوی بیرون و درون روند ریسمان بگذارد.

 

یک راه دیگر برای فرستادن داده به یک ریسمان این است که به جای اینکه یک متد static به ریسمان بدهیم یک متد نمونه (instance) به ریسمان بدهیم. با این کار میتوان با استفاده از خصوصیات شیء نمونه به ریسمان بگوییم چه کاری باید انجام دهد. در اینجا را یک بار دیگر مینویسیم :

class ThreadTest {

  bool upper;

  static void Main() {

    ThreadTest instance1 = new ThreadTest();

    instance1.upper = true;

    Thread t = new Thread (instance1.Go);

    t.Start();

    ThreadTest instance2 = new ThreadTest();

    instance2.Go(); // Main thread – runs with upper=false

  }

  void Go() { Console.WriteLine (upper ? "HELLO!" : "hello!"); }
