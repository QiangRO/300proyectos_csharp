مدیریت استثناء ( Exception Handling )

وقتیکه یک  ریسمان ساخته شده و اجرایش شروع شود دیگر بلوک try/catch/finally که مخصوص مدیریت استثناست کارایی خود را ندارد. برنامه زیر را در نظر بگیرید :

public static void Main() {

  try {

    new Thread (Go).Start();

  }

  catch (Exception ex) {

    // We'll never get here!

    Console.WriteLine ("Exception!");

  }

  static void Go() { throw null; }

}

 

عبارت کنترلی try/catch این مثال بی تاثیر است ، و ریسمان ساخته شدۀ جدید با یک NullReferenceException اسباب زحمت خواهد شد. این بدان معنیست که یک ریسمان مسیر اجرای مستقلی دارد. چاره این مشکل این است که متدهای مدخل ریسمان مدیر استثناء مخصوص خودشان را داشته باشند :

 

public static void Main() {

  new Thread (Go).Start();

}

static void Go() {

  try {

    ...

    throw null; // this exception will get caught below

    ...

  }

  catch (Exception ex) {

        Typically log the exception, and/or signal another thread

        that we've come unstuck

  ...

}

از .NET 2.0 به بعد ، یک استثناء مدیریت نشده در هر یک از ریسمانهای برنامه ، کل برنامه را میبندد و خاتمه میدهد، بنابراین نباید استثناءها را نادیده گرفت. بنابراین در متد هر یک از ریسمانها باید یک بلوک try/catch بگذاریم . حداقل برای برنامه های تولیدی که قصد فروش و به کار بردن آنها را داریم باید این کار را انجام دهیم تا برنامه با یک استثناء مدیریت نشده بسته نشود. به ویژه برای برنامه نویسان برنامه های فرمی ویندوزی که معمولاً از یک مدیر استثناء "global"  استفاده میکنند ، مانند زیر :

using System;

using System.Threading;

using System.Windows.Forms;

static class Program {

  static void Main() {

    Application.ThreadException += HandleError;

    Application.Run (new MainForm());

  }

  static void HandleError (object sender, ThreadExceptionEventArgs e) {

        Log exception, then either exit the app or continue...

  }

}

زمانی که یک استثناء رخ میدهد رویداید Application.ThreadException رخ میدهد که آخرین تابع صدا زده شده بعنوان نتیجه یک پیام ویندوزی است ( مثلاً یک پیام keyboard ، mouse ، یا paint ) ، تقریباً برای تمام کد یک برنامه معمولی فرمی ویندوزی اینگونه است.از آنجا که این شیوه یعنی مدیریت استثناءها با یک مدیر مرکزی موثر عمل میکند ، میتواند خیال هر کس را به اشتباه در مورد یک حس امنیتی نادرست آسوده کند.استثناءهای مربوط به ریسمانهای کارگر مثال خوبی از استثناءهایی هستند که توسط ApplicationThreadException مدیریت نمیشوند. ( کد درون متد main نیز نمونه دیگریست ، در متد سازنده (constructor) اصلی فرم ، زیرا قبل از شروع حلقه پیام ویندوز – windows message – اجرا میشود ) .

.NET framework برای مدیریت استثناء عمومی یک رویداد سطح پایینتر دارد : AppDomain.UnhandledException

این رویداد هر زمان که استثناء مدیریت نشده ای در هر یک از ریسمانها اتفاق بیافتد ، رخ میدهد. در هر نوع برنامه ای ، خواه رابط کاربری داشته باشد خواه نداشته باشد.به هر حال با اینکه این رویداد آخرین مکانیزم چاره ساز و خوب را برای ثبت استثناءهای کنترل نشده ارائه میدهد ، از بسته شدن و shut down شدن برنامه جلوگیری نمیکند، همچنین پنجره دیالوگ استثناءهای مدیریت نشده را در .NET پشتیبانی نمیکند.

 

در برنامه های واقعی مصرفی ، مدیریت استثناء صریح بر روی تمام متدهای ورودی ریسمان لازم است. ممکن است کسی برای این کار از یک wrapper یا یک کلاس helper مانند یک BackgroundWorker استفاده کند(توضیح بیشتر در بخش 3 )
