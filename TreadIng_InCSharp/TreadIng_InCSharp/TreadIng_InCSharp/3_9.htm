تایمرها ( Timers )

ساده ترین راه برای اجرای یک متد بصورت مکرر و در زمانهای معین ، استفاده از یک timer است (از قبیل کلاس Timer که در فضای نام System.Threading ارائه شده است ) . مزیت threading timer برتر از thread pool است زیرا میتوان تایمرهای فراوانی تولید کرد بدون اینکه سرباری ناشی از فراوانی ریسمانها ایجاد شود . Timer انصافاً یک کلاس ساده است که شامل یک سازنده و فقط دو متد است .

public sealed class Timer : MarshalByRefObject, IDisposable

{

  public Timer (TimerCallback tick, object state, 1st, subsequent);

  public bool Change (1st, subsequent); // To change the interval

  public void Dispose(); // To kill the timer

}

1st = time to the first tick in milliseconds or a TimeSpan

subsequent = subsequent intervals in milliseconds or a TimeSpan

  (use Timeout.Infinite for a one-off callback)

 

در مثال زیر ، یک timer متد Tick را فراخوانی میکند که هر 6 ثانیه یک بار مینویسد "tick…" تا زمانیکه کاربر کلید Enter را بزند :

using System;

using System.Threading;

class Program {

  static void Main() {

    Timer tmr = new Timer (Tick, "tick...", 5000, 1000);

    Console.ReadLine();

    tmr.Dispose(); // End the timer

  }

  static void Tick (object data) {

    // This runs on a pooled thread

    Console.WriteLine (data); // Writes "tick..."

  }

}

.NET framework کلاس timer دیگری با همین نام در فضای نام System.Timers دارد که بهتر از System.Threading.Timer عمل میکند و در صورت استفاده در یک استخر ریسمان یکسان آسودگی بیشتری فراهم میکند ولی هر دو موتور یکسانی دارند . در زیر برخی از ویژگیهای اضافی آنرا می بینیم :

 

  • یک پیاده سازی از نوع کلاس Component که به آن اجازه میدهد در Visual Studio Designer استفاده شود
  • یک ویژگی به نام Interval به جای متد Change
  • یک رویداد به نام Elapsed به جای callback delegate
  • یک ویژگی به نام Enabled برای شروع و توقف تایمر (مقدار پیش فرض آن false است)
  • متدهای Start و Stop برای وقتی که استفاده از Enabled گیج کننده باشد
  • یک پرچم به نام AutoReset برای نشان دادن یک رویداد بازگشتی (مقدار پیش فرض آن true است)

 

در زیر مثالی را می بینیم :

using System;

using System.Timers; // Timers namespace rather than Threading

class SystemTimer {

  static void Main() {

    Timer tmr = new Timer(); // Doesn't require any args

    tmr.Interval = 500;

    tmr.Elapsed += tmr_Elapsed; // Uses an event instead of a delegate

    tmr.Start(); // Start the timer

    Console.ReadLine();

    tmr.Stop(); // Pause the timer

    Console.ReadLine();

    tmr.Start(); // Resume the timer

    Console.ReadLine();

    tmr.Dispose(); // Permanently stop the timer

  }

  static void tmr_Elapsed (object sender, EventArgs e) {

    Console.WriteLine ("Tick");

  }

}

.NET framework هنوز یک timer سوم دیگری را نیز ارائه میدهد که در فضای نام System.Windows.Forms است . با اینکه در ظاهر شبیه System.Timers.Timer است ولی از ریشه و اساساً از نظر کارکرد و وظیفه با آن فرق دارد . یک Windows Forms Timer از استخر ریسمان استفاده نمیکند ، در عوض همیشه رویداد "Tick" خود را در همان ریسمانی که آن timer را ساخته ، برانگیخته می کند . حال فرض کنید این ریسمان ، ریسمان اصلی "main" برنامه باشد که مسوول ساختن نمونه های همه فرمها و کنترلهای آنها در یک برنامه فرمی ویندوز است ، با این فرض اداره کنندۀ تایمر (timer’s event handler) میتواند بدون اینکه thread-safety را نقض کند یا نگرانی از apartment-threading با فرمها و کنترلها فعل و انفعال و تبادل کند . Control.Invoke نیز احتیاج نیست .

 

یک windows forms timer برای کارهایی طراحی شده است که شامل به روز رسانی رابط کاربری هستند و یا کارهایی که سریع اجرا میشوند . اجرای سریع در اینجا مهم است زیرا رویداد tick در ریسمان اصلی برنامه فراخوانی میشود که اگر زمانبر و محدود کننده کارایی باشد رابط کاربری را بی مسوولیت میکند (یعنی برنامه جواب نمیدهد یا کند جواب میدهد) .
