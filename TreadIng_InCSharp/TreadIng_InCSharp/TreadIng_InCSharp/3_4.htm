کارگر پس زمینه ( BackgroundWorker )

کلاس BackgroundWorker یک کلاس کمکی در فضای نام System.ComponentModel است که برای مدیریت ریسمانهای کارگر استفاده میشود . این کلاس ویژگیهای زیر را ارائه میدهد :

 

  • یک پرچم به نام "cancel" برای اعمال سیگنال به ریسمان کارگر تا پایان یابد (بدون استفاده از Abort)
  • یک قرارداد (protocol) استاندارد به منظور گزارش پیشرفت ، تکمیل و یا انصراف
  • یک پیاده سازی از واسط IComponent  که به آن اجازه میدهد در Visual Studio Designer قرار گیرد
  • مدیریت استثناء بر روی ریسمان کارگر
  • قابلیت به روز رسانی کنترلهای فرمی ویندوز با پیشرفت یا تکمیل ریسمانهای کارگر

 

مخصوصاً دو ویژگی آخر بسیار مفید هستند . که معنی آن این است که مجبور نیستیم در متد کارگر بلاک try/catch قرار دهیم و همچنین میتوانیم کنترلهای فرمی ویندوز را بدون احتیاج به فراخوانی متد Control.Invoke به روز کنیم "update" .

 

کلاس BackgroundWorker از یک thread-pool یا استخر ریسمان استفاده می کند که ریسمانها را بازسازی میکند و از ساخته شدن دوباره ریسمانها برای کارهای جدید جلوگیری می کند که معنی آن این است که هرگز لازم نیست متد Abort را بر یک ریسمان BackgroundWorker صدا کنیم .

 

برای استفاده از BackgroundWorker تنها کافیست قدمهای کوچک زیر را برداریم :

·         ساختن یک نمونه از شیء BackgroundWorker و اداره کردن رویداد DoWork .

·         فراخوانی متد RunWorkerAsync با یک پارامتر اختیاری

 

با این کار آنرا بکار برده ایم . هر پارامتری که به متد RunWorkerAsync پاس داده شود از طریق ویژگی Argument پارامتر رویداد (e.Argument) به کنترل کننده رویداد DoWork (DoWork event handler ) فرستاده میشود (forward) . در اینجا مثالی را می بینیم :

class Program {

  static BackgroundWorker bw = new BackgroundWorker();

  static void Main() {

    bw.DoWork += bw_DoWork;

    bw.RunWorkerAsync ("Message to worker");

    Console.ReadLine();

  }

  static void bw_DoWork (object sender, DoWorkEventArgs e) {

    // This is called on the worker thread

    Console.WriteLine (e.Argument); // writes "Message to worker"

    // Perform time-consuming task...

  }

کلاس BackgroundWorker همچنین یک رویداد به نام RunWorkerCompleted دارد که پس از اینکه اداره کننده رویداد DoWork یعنی "DoWork event handler" کارش را انجام داد ، رخ میدهد . اداره کردن رویداد RunWorkerCompleted اجباری نیست ولی میتوان این کار را برای فهمیدن اینکه در DoWork استثناء روی داده است یا نه انجام داد . بنابراین کد داخل کنترل کننده رویداد RunWorkerCompleted قادر است کنترلهای فرمی ویندوز را بدون هیچگونه marshalling صریح به روز رسانی کند که این کار را کد داخل کنترل کننده رویداد DoWork نمیتواند انجام دهد .

 

برای پشتیبانی از گزارش گیری پیشرفت :

  • مقدار ویژگی WorkerReportsProgress را true می کنیم
  • در داخل کنترل کننده رویداد DoWork به طور مکرر و در زمانهای معینی (periodically) متد ReportProgress را با  یک مقدار "percentage complete" و در صورت دلخواه با یک شیء user-state فراخوانی می کنیم
  • رویداد ProgressChanged را اداره می کنیم ، یعنی مقدار ویژگی ProgressPercentage پارامتر آن را بررسی میکنیم

 

کد داخل کنترل کننده  رویداد ProgressChanged مانند همان مکانیزم RunWorkerCompleted میتواند با کنترلهای UI (رابط کاربری) فعل و انفعال و تبادل کند . چنین چیزی به طور معمول زمانی اتفاق می افتد که یک progress bar را به روز یا update می کنیم .

 

برای پشتیبانی انصراف (cancellation) :

  • مقدار ویژگی WorkerSupportsCancellation را true می کنیم
  • در داخل کنترل کننده رویداد DoWork به طور مکرر و در زمانهای معینی مقدار ویژگی CancellationPending را بررسی و چک می کنیم . هر وقت true شد ویژگی Cancel پارامتر رویداد را true می کنیم و return می کنیم . (ریسمان کارگر میتواند بدون تنظیم یا مقداردهی ویژگی CancellationPending مقدار Cancel را true کند و خارج شود ، در صورتیکه احساس کند کار خیلی مشکل شده و نمیتواند ادامه دهد )
  • برای درخواست انصراف متد CancelAsync را فراخوانی می کنیم

 

در اینجا مثالی را مشاهده می کنیم که تمام مباحث فوق را بکار می برد :

using System;

using System.Threading;

using System.ComponentModel;

class Program {

  static BackgroundWorker bw;

  static void Main() {

    bw = new BackgroundWorker();

    bw.WorkerReportsProgress = true;

    bw.WorkerSupportsCancellation = true;

    bw.DoWork += bw_DoWork;

    bw.ProgressChanged += bw_ProgressChanged;

    bw.RunWorkerCompleted += bw_RunWorkerCompleted;

    bw.RunWorkerAsync ("Hello to worker");

    Console.WriteLine ("Press Enter in the next 5 seconds to cancel");

    Console.ReadLine();

    if (bw.IsBusy) bw.CancelAsync();

    Console.ReadLine();

  }

  static void bw_DoWork (object sender, DoWorkEventArgs e) {

    for (int i = 0; i <= 100; i += 20) {

      if (bw.CancellationPending) {

        e.Cancel = true;

        return;

      }

      bw.ReportProgress (i);

      Thread.Sleep (1000);

    }

    e.Result = 123; // This gets passed to RunWorkerCopmleted

  }

  static void bw_RunWorkerCompleted (object sender,

  RunWorkerCompletedEventArgs e) {

    if (e.Cancelled)

      Console.WriteLine ("You cancelled!");

    else if (e.Error != null)

      Console.WriteLine ("Worker exception: " + e.Error.ToString());

    else

      Console.WriteLine ("Complete - " + e.Result); // from DoWork

  }

  static void bw_ProgressChanged (object sender,

  ProgressChangedEventArgs e) {

    Console.WriteLine ("Reached " + e.ProgressPercentage + "%");

  }

}

تعریف زیر کلاس از کلاس BackgroundWorker  ( Subclassing BackgroundWorker )

 

BachgroundWorker یک کلاس sealed نیست (مفاهیم شیء گرایی) و یک متد مجازی به نام OnDoWork ارائه میدهد که الگوی دیگری از استفاده این کلاس را ارائه میدهد . گاهی اوقات یک متد مینویسیم که بالقوه و ذاتاً زمان اجرای طولانی دارد ، شاید بخواهیم به جای آن ویا در کنار آن ورژن دیگری از چنین متدی را بنویسیم که یک زیر کلاس از نوع BackgroundWorker برمیگرداند و از قبل برای انجام کار بصورت اسنکرون یا غیر همزمان تنظیم شده است . برای استفاده از چنین کلاسی فقط کافیست رویدادهای RunWorkerCompleted و ProgressChanged را اداره کند . برای مثال فرض کنید یک متد زمانگیر به نام GetFinancialTotals نوشته ایم :

public class Client {

  Dictionary <string,int> GetFinancialTotals (int foo, int bar) { ... }

  ...

}

 

میتوانی به جای آن بصورت زیر عمل کنیم :

 

public class Client {

  public FinancialWorker GetFinancialTotalsBackground (int foo, int bar) {

    return new FinancialWorker (foo, bar);

  }

}

public class FinancialWorker : BackgroundWorker {

  public Dictionary <string,int> Result; // We can add typed fields.

  public volatile int Foo, Bar; // We could even expose them

  // via properties with locks!

  public FinancialWorker() {

    WorkerReportsProgress = true;

    WorkerSupportsCancellation = true;

  }

  public FinancialWorker (int foo, int bar) : this() {

    this.Foo = foo; this.Bar = bar;

  }

  protected override void OnDoWork (DoWorkEventArgs e) {

    ReportProgress (0, "Working hard on this report...");

         Initialize financial report data

    while (!finished report ) {

      if (CancellationPending) {

        e.Cancel = true;

        return;

      }

             Perform another calculation step

      ReportProgress (percentCompleteCalc, "Getting there...");

    }

    ReportProgress (100, "Done!");

    e.Result = Result = completed report data;

  }

}

هر کسی که متد GetFinancialTotalsBackground را فراخوانی کند ، یک شیء FinanciaWorker دریافت خواهد کرد (این مثالی از مدیریت BackgroundWorker در یک کاربرد واقعی است) . با این کلاس میتوان پیشرفت را گزارش داد و اعمال انصراف کرد ، همچنین بدون استفاده از Control.Invoke با برنامه های فرمی ویندوز سازگار است . این کلاس همچنین در استثناءها مدیریت شده است و از یک قرارداد ساده نیز استفاده می کند (هر کس میتواند از BackgroundWorker استفاده کند ) .

بکار بردن کلاس BackgroundWorker بدین صورت موثرتر از شیوه استفاده قدیمی قبلی است که یک الگوی اسنکرون مبتنی بر رویداد است "event-based asynchronous pattern" .
