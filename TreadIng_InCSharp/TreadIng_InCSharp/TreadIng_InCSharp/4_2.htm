اتمی عمل کردن و قرار گرفتن در کنار هم با الگوی همزمانی ( Atomicity and Interlocked )

یک عبارت "statement" بصورت اتومیک است اگر بصورت یک دستورالعمل غیرقابل تقسیم محض اجرا شود . اتمی عمل کردن محض ، هر گونه احتمال پیش دستی به قصد تصرف را رد میکند . در C# ، یک عمل خواندن یا اختصاص ساده بر یک فیلد 32 یبتی یا کوچکتر ، عمل اتومیک است (البته با فرض 32 بیتی بودن CPU) . عملیات بر فیلدهای بزرگتر ، اتومیک نیستند . همین طور عبارات مرکب از بیش از یک عمل خواندن یا نوشتن ، نیز اتومیک نیستند :

 

class Atomicity {

  static int x, y;

  static long z;

  static void Test() {

    long myLocal;

    x = 3; // Atomic

    z = 3; // Non-atomic (z is 64 bits)

    myLocal = z; // Non-atomic (z is 64 bits)

    y += x; // Non-atomic (read AND write operation)

    x++; // Non-atomic (read AND write operation)

  }

}

خواندن و نوشتن فیلدهای 64 بیتی در CPU های 32 بیتی اتومیک نیستند زیرا در آنها با دو فضای حافظۀ 32 بیتی مجزا درگیر هستیم . اگر ریسمان A در حال خواندن یک فیلد 64 بیتی باشد و در همین حال ریسمان B در حال update همان فیلد باشد ، امکان دارد حاصل ریسمان A با یک ترکیب بیتی از مقادیر قدیم و جدید این فیلد بشود .

 

عملگرهای یکتایی از نوع x++ احتیاج دارند که در ابتدا یک متغیر را خوانده ، سپس آنرا پردازش کرده و سپس دوباره آنرا بنویسند . کلاس زیر را در نظر بگیرید :

class ThreadUnsafe {

  static int x = 1000;

  static void Go () { for (int i = 0; i < 100; i++) x--; }

}

 

شاید توقع داشته باشید که اگر 10 ریسمان به طور همزمان متد Go را اجرا کنند ، مقدار نهایی x صفر شود . به هر حال چنین چیزی گارانتی نمیشود زیرا همیشه این امکان وجود دارد که یک ریسمان مابین بازیابی مقدار کنونی x ، کاهش آن و نوشتن دوبارۀ آن ریسمان دیگری را قبضه کند (که چنین چیزی باعث میشود که یک مقدار out-of-data بوجود بیاید ، منظور مقدار پایانی برای x است)

 

یک چاره برای حل چنین مشکلاتی این است که اعمال غیر اتومیک را در یک قفل بپوشانیم . در واقع عمل قفلگذاری مکانیزم اتومی عمل کردن را شبیه سازی میکند . به این همه ، کلاس Interlocked راه حل ساده تر و سریعتری برای اعمال اتومیک ساده ارائه میدهد :

class Program {

  static long sum;

  static void Main() { // sum

    // Simple increment/decrement operations:

    Interlocked.Increment (ref sum); // 1

    Interlocked.Decrement (ref sum); // 0

    // Add/subtract a value:

    Interlocked.Add (ref sum, 3); // 3

    // Read a 64-bit field:

    Console.WriteLine (Interlocked.Read (ref sum)); // 3

    // Write a 64-bit field while reading previous value:

    // (This prints "3" while updating sum to 10)

    Console.WriteLine (Interlocked.Exchange (ref sum, 10)); // 10

    // Update a field only if it matches a certain value (10):

    Interlocked.CompareExchange (ref sum, 123, 10); // 123

  }

}

استفاده از Interlocked بسیار موثرتر از بکار بردن یک قفل است ، زیرا هرگز بلوکه نمیشود و لذا سرباری از جهت خروج موقت از زمانبندی متوجه آن نیست .

Interlocked همچنین در گسترۀ چندین پردازه نیز درست عمل میکند .بر خلاف lock که فقط در بین ریسمانهای پردازه جاری موثر است . یک مثال برای جایی که Interlocked میتواند موثر عمل کند در عمل خواندن و نوشتن در حافظۀ اشتراکی "shared memory" است .
